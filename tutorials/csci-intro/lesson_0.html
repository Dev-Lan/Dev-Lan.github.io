Lesson_0 : Do as I say computer!

Finally we get to begin our journey of ordering around computers to do our bidding. In fact our electronic servants only ever perform tasks if some person has programmed that behavior into them. What's more, they carry out these tasks exactly as they have been instructed to. While at its core this is a great feature, at first it can seem like you are trying to get help from Amelia Bedelia.

While learning this is at worst a annoying inconvenience, but if you ever end up doing more serious projects with your code the results may be more drastic.

So even though the computer will always do exactly as you say, sometimes the results can surprise you. In particular work with artificial intelligence is interesting since often the instructions you are giving your computer are not to do a specific task, but rather to learn how to do that task. This distinction has generated some concern about what will happen if we teach a computer to outthink us. In my (and many others more esteemed than me) think that this is a valid concern. That said this is a fairly nuanced topic to delve into here, but suffice it to say that a panicky fear of a robotic uprising is not the appropriate response.

When you first start programming, your biggest obstacle in communicating with the computer will not be accidentally creating a Skynet system of robotic overlords that rules over humans with an iron fist. Instead it will be the much more mundane task of dealing with syntax. Syntax is the set of rules that defines how you give commands to your computer. I'm no expert in language theory, so informally (read likely pedantically incorrect) I think of syntax like grammar in human languages, except the computer will tolerate no mistakes. It's like if your english teacher read your 10 page report on pterodactyl grooming behavior, discovered you used a comma when it should've been a semicolon and as a result gave you a 0% on the project. If you have incorrect syntax and you try to run your code it will fail and give you a syntax error.

We are going to begin by examining in closer detail the code from the last lesson. 

# Copy Me!
size(400, 400)
background(0, 0, 0)
fill(255, 255, 0)
arc(200, 200, 360, 360, QUARTER_PI, 7 * QUARTER_PI, PIE)

Line 1 is a comment. Comments are placed in code to help humans understand the code. The computer will completely ignore comments. In Python to make a comment you simply use the number sign #, or hashtag for all you twitter lovers. Anything that follows a # will be ignored by the computer. The last four lines are all function calls to functions that are provided by Processing. A function is a chunk of code that you want to use multiple times. To call a function in Python you have to use the following syntax. 

functionName(firstArgument, secondArgument, ..., lastArgument)

Arguments are the things that you pass into your function. A function can be defined to take any number of arguments, including zero. When you call your function you have to pass in the correct number of arguments or else you will suffer a syntax error. If you have zero arguments you still need an open and closing parenthesis, but there will be nothing between them. 

# You can comment / uncomment code with "command + /"" on Mac 
size(400, 400)
# background(0, 0, 0)
# fill(255, 255, 0)
# arc(200, 200, 360, 360, QUARTER_PI, 7 * QUARTER_PI, PIE)

Here I have commented out everything except line 2. This line calls the processing function "size". Here the function name is size. The first argument of 400 specifies how many pixels wide the window will be, and the second argument, which also happens to be 400 does the same thing same thing in height. Try running the code above. Change the arguments to different values. What do you expect to see if you call size(200, 800). What do you actually see? What if you have multiple calls to size?

All of the functions provided by Processing are described on their reference page. You should refer to this after my explanations for more details. 

# You can comment / uncomment code with "ctrl + /"" on Windows 
size(400, 400) # sets size of window
background(0, 0, 0)
# fill(255, 255, 0)
# arc(200, 200, 360, 360, QUARTER_PI, 7 * QUARTER_PI, PIE)

If you uncomment line 3 and run your code again, you will can see that background will change the background color of the window. Color is typically represented with three values. You can leave it at that for now, but if your are really interested you can look at Processing's color tutorial. 

# blah blah blah blah
size(400, 400) # sets size of window
background(0, 0, 0) # sets background color of window to black
fill(255, 255, 0)
# arc(200, 200, 360, 360, QUARTER_PI, 7 * QUARTER_PI, PIE)

If you uncomment line 4 and run your code again you will see that there is no change in the output you see. Some code you will run will have no apparent effect on their own, but are doing important things behind the scenes. In this case fill changes the color that will be used for subsequent shapes drawn. This operation is like picking a color in your favorite paint program. 

# The sauce makes the goose more succulent
size(400, 400) # sets size of window
background(0, 0, 0) # sets background color of window to black
fill(255, 255, 0) # sets color of subsequent shapes to yellow
arc(200, 200, 360, 360, QUARTER_PI, 7 * QUARTER_PI, PIE) # draws arc shape
